这份指南对 nginx 进行了基本的介绍并且描述了一些可以用 nginx 来做的简单任务。 这里假定读者的机器上已经安装了 nginx。 如果还没有安装，见安装 nginx页面。 这份指南描述了如何去启动和停止 nginx，以及如何重新加载它的配置，解释了配置文件的结构并且描述了 如何去设置 nginx 从而提供静态内容，如何去配置 nginx 使其作为一个代理服务器，还有如何用一个 FastCGI 应用程序去连接 nginx。

nginx 有一个 master 进程和几个 worker 进程。 master 进程的主要任务是读取和评估配置，以及维护 worker 进程。 worker 进程才助理真正的请求。 nginx 在 worker 进程间 采用基于事件的模型和操作系统依赖的机制去有效地分发请求。 worker 进程的数量被定义在配置文件中并且可能对一个给定的配置中来说是固定的或者自动的调整为可用的 cpu 核数 （见worker 进程）。

在配置文件中决定了 nginx 和它的模块的工作方式。 默认情况下，配置文件被命名为nginx.conf并被放在 /usr/local/nginx/conf， /etc/nginx, 或 /usr/local/etc/nginx目录下。

启动、停止和重新加载配置

运行可执行文件去启动 nginx。 一旦 nginx 被启动了，它可以通过调用带有-s参数的可执行程序来控制。 使用如下语法： 

nginx -s signal

其中的 signal 可以是下面这些值。

  • stop — 快速关闭
  • quit — 优雅的关闭
  • reload — 重新加载配置文件
  • reopen — 重新打开日志文件

例如，如果想要等待 worker 进程服务完当前的请求再关闭 nginx 进程，可以执行下面的命令： 

nginx -s quit

这个命令应当在启动当前 nginx 的用户下被执行

除非重新加载配置文件的命令被发送到 nginx 或 nginx 被重启了，否则配置文件中 所做的修改不会被应用。 为了重新加载配置文件，请执行： 

nginx -s reload

一旦 master 进程收到了重载配置文件的信号， 它会去检查新的配置文件的语法合法性并且试图去应用里面提供的配置。 如果成功了，这个 master 进程会启动新的 worker 进程并且发送消息给老的 worker 进程，请求他们关闭。 否则，master 进程回滚修改并且继续用旧的配置进行工作。 旧的 worker 进程收到命令去关闭，停止接受新的连接并且继续服务当前的请求直到 所有这些请求被服务完毕。 然后，旧的 worker 进程终止运行。

也可以通过 Unix 的帮助工具比如 kill 工具去发送一个信号给 nginx 进程。 在这种情况下，一个信号通过一个给定的进程ID被直接发送给对应的进程。 nginx master 进程的进程ID默认写在 /usr/local/nginx/logs 或 /var/run目录下的 nginx.pid。 例如，如果 master 进程的ID是 1628， 通过发送 QUIT 信号使得 nginx 优雅的被关闭，可以执行 

kill -s QUIT 1628

为了得到一个正在运行的 nginx 进程列表，ps 工具可能被用到，例如下面这种方式： 

ps -ax | grep nginx

关于发送信号给 nginx 的更多信息，见 控制 nginx。

配置文件的结构

nginx 由模块组成，被在配置文件中指定的指令控制。 指令被划分为简单指令和块指令。 一个简单指令由名字和以空格分隔，分号结尾(;)的参数组成， 一个块指令作为一个简单指令有相同的结构， 但它以一组用大括号({ 和 })包围起来的额外指令结尾 而不是分号。 如果一个块指令的大括号内部还有其他的指令， 它就被称为一个 context（例如： events, http, server, 和 location）。

配置文件中被放在任何 context 之外的指令被认为在 main context中。 events 和 http 指令 驻留在 main context中, server 驻留在 http中, location 在 server中。

一行中 # 号后面的部分被视为注释。

提供静态内容

web 服务器的一个重要任务就是提供文件 （比如图片或静态的 HTML 页面）。 在这里你将实现一个例子，对于不同的请求 将提供来自不同本地目录 /data/www (可能包含 HTML 文件) 和 /data/images (包含图片) 中的文件。 这将需要编辑配置文件并在带有两个 location块 的http块 中设置一个server块。

首先，创建 /data/www 文件夹并放置一个带有任何文件内容的 index.html文件进去。 同样的，创建一个 /data/images 文件夹并放置一些图片文件进去。

下一步，打开配置文件。 默认的配置文件已经包含了几个 server 块的例子， 其中大多数被注释掉了。 现在注释掉所有的这些块并开始一个新的 server 块： 

http {
    server {
    }
}

一般情况下，配置文件可能包含几个 server 块， 这些块通过他们监听的端口和 服务器名称 来进行 区分。 一旦 nginx 决定了哪个 server 去处理请求， 它就会测试在请求头部中指定的 URI 与 server 块中定义的 location 指令的参数。

添加下面的 location 块到 server 块中： 

location / {
    root /data/www;
}

这个 location 块指定了“/” 前缀， 这个前缀将与请求中的 URI 进行比较。 匹配了请求后，这个 URI 将被添加到 在 root 指令 中指定的路径。 在这里，它是 /data/www， 在本地文件系统中去构造出请求文件的路径。 如果有几个匹配的 location 块，nginx 选择一个最长前缀匹配。 上面的 location 块提供了长度为一的最短前缀， 仅当所有其他的 location 块都没有匹配时， 这个块才会被使用。

下一步，添加第二个 location 块： 

location /images/ {
    root /data;
}

它将匹配一个以 /images 开头的请求（ location / 也会匹配那些请求，但匹配的前缀更短）。

最终的 server 块的配置看起来就像这样： 

server {
    location / {
        root /data/www;
    }

    location /images/ {
        root /data;
    }
}

这已经是一个正在工作的服务器配置，它监听在标准的 80 端口上， 并且它在本地机器上的 http://localhost/ 也是可以访问的。 为了响应以 /images 开头的 URI请求， 服务器将从 /data/images 目录下发送文件。 例如，为了响应 http://localhost/images/example.png 请求，nginx 将 发送 /data/images/example.png 文件。 如果这个文件不存在，nginx 将发送一个指示为404错误的响应。 不以 /images 开头的URI请求将映射到 /data/www 目录下。 例如，为了响应 http://localhost/some/example.html 请求，nginx 将 发送 /data/www/some/example.html 文件。

为了应用一个新的配置，如果 nginx 还没有启动就启动它， 或者发送 reload 信号给 nginx 的 master 进程， 通过执行： 

nginx -s reload

倘使不按预期工作，你可能需要 在 /usr/local/nginx/logs 或 /var/log/nginx 目录下 的 access.log 和 error.log 文件中查出错误原因。

设置一个简单的代理服务器

nginx 最常见的用法之一就是设置它为一个代理服务器， 这意味着一个收到请求的服务器将会将请求转发到代理服务器， 并从代理服务器那里取回响应，并发送他们给客户。

我们将配置一个基础的代理服务器，它将用来自本地目录的文件提供图片请求服务 并发送所有其他请求给代理服务器。 在这个例子中，两个服务器都将定义在一个单一的 nginx 实例下。

首先，使用下面的内容，添加一个或多个 server 块给 nginx 的配置文件， 从而去定义代理服务器。 

server {
    listen 8080;
    root /data/up1;

    location / {
    }
}

这是一个监听在8080端口的简单服务器， （上一次，因为使用了标准的80端口， 并没有指定 listen 指令） 并且映射所有的请求到 本地文件系统的 /data/up1 目录。 创建这个目录并将 index.html 文件放进去。 记住 root 指令被放在 server context。 当被选择去服务一个请求的 location 块没有 自己的 root 指令时， 这个 root 指令就会被使用。

下一步，使用上一个部分的服务器配置并修改它使得它成为一个代理服务器的配置。 在第一个 location 块中，放置带有 协议，名字和在参数中指定的代理服务器的端口 的 proxy_pass 指令 (在我们的例子中，它是 http://localhost:8080): 

server {
    location / {
        proxy_pass http://localhost:8080;
    }

    location /images/ {
        root /data;
    }
}

我们将修改第二个 location 块， 它当前将以 /images 作为前缀的请求映射到 /data/images 目录下的文件。 为了使得它能够匹配带有一般文件扩展的图片请求。 修改后的 location 块看起来就像这样： 

location ~ \.(gif|jpg|png)$ {
    root /data/images;
}

参数是一个能够匹配所有以 .gif, .jpg, 或 .png 结尾的 URI 的正则表达式。 一个正则表达式应该以 ~ 开头。 相应的请求将会被映射到 /data/images 目录。

当 nginx 选择一个 location 块去服务一个请求时， 它首先会检查指定前缀的location 指令，并记住带有最长前缀的 location，然后检查正则表达式。 如果有一个正则表达式匹配，nginx 选择这个 location，否则， 它选择更早之前记住的 location。

最终的代理服务器配置看起来就像这样： 

server {
    location / {
        proxy_pass http://localhost:8080/;
    }

    location ~ \.(gif|jpg|png)$ {
        root /data/images;
    }
}

这个服务器将过滤掉以 .gif, .jpg, 或 .png 结尾的请求， 并映射他们到 /data/images 目录（ 通过添加 URI 到 root 指令的参数） 并传递其他所有请求到上面配置的代理服务器。

为了应用一个新的配置，像前面的部分所描述的那样发送 reload 信号到 nginx。

有很多可能被用于进一步配置代理连接的更多指令。

设置 FastCGI 代理

nginx 可以被用于到 FastCGI 服务器的常规请求， 它运行着由各种各样的框架和编程语言比如 PHP 构建的应用程序。

与 FastCGI 服务器一同工作的最基础的 nginx 配置包括使用 fastcgi_pass指令 而不是 proxy_pass 指令，以及设置传递给 FastCGI 服务器的参数的 fastcgi_param 指令。 假定 FastCGI 服务器在 localhost:9000 上是可以访问的。 将上一个部分的代理配置拿过来作为基础，将 proxy_pass 指令 用 fastcgi_pass 指令进行替换，并改变到 localhost:9000 的参数。 在 PHP 里，SCRIPT_FILENAME 参数被用于决定脚本名，而 QUERY_STRING 参数被用于传递请求参数。 最终的配置就是： 

server {
    location / {
        fastcgi_pass  localhost:9000;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        fastcgi_param QUERY_STRING    $query_string;
    }

    location ~ \.(gif|jpg|png)$ {
        root /data/images;
    }
}

这将设置一个服务器，该服务器将通过 FastCGI 协议将除静态图像请求之外的所有请求路由到在 localhost:9000 上运行的代理服务器。
