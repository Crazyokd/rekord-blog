生命周期的核心问题是：

> 引用到底能活多久？

生命周期不是 Rust 为了折磨新手发明的语法，而是借用系统必须回答的问题。

# 生命周期解决什么问题

生命周期解决的是引用有效性。

引用必须指向有效的值。如果值已经被释放，引用就会变成悬空引用。

Rust 不允许悬空引用。

```rust
fn main() {
    let r;

    {
        let s = String::from("hello");
        r = &s;
    }

    // println!("{}", r); // 错误：s 已经被释放
}
```

`s` 在内部作用域结束时被释放，`r` 不能继续引用它。

这就是生命周期要解决的问题：

> 引用不能比它指向的值活得更久。

# 编译器通常能推导

大多数时候，你不需要手写生命周期。

```rust
fn first_word(s: &str) -> &str {
    let bytes = s.as_bytes();

    for (i, &item) in bytes.iter().enumerate() {
        if item == b' ' {
            return &s[0..i];
        }
    }

    s
}
```

这个函数没有写生命周期，但可以通过编译。

因为编译器能看出来：

- 参数是一个 `&str`
- 返回值也是从这个参数里切出来的 `&str`
- 返回引用的有效性依赖参数引用

这种情况不需要手写。

# 什么时候需要手写生命周期

当函数有多个引用参数，并且返回值也是引用时，编译器可能无法判断返回值来自哪个参数。

```rust
fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

这段代码不能通过编译。

因为返回值可能来自 `x`，也可能来自 `y`。编译器不知道返回引用应该和哪个参数绑定。

所以要写成：

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

这告诉编译器：

> 返回值的生命周期，不能超过 `x` 和 `y` 中较短的那个。

生命周期标注不是改变代码行为，而是把引用关系说清楚。

# 生命周期省略规则

Rust 有生命周期省略规则，所以很多函数不用手写生命周期。

常见规则可以先记三条：

1. 每个引用参数都有自己的生命周期。
2. 如果只有一个引用参数，返回引用默认和它相同。
3. 如果方法参数里有 `&self` 或 `&mut self`，返回引用默认和 `self` 相同。

例如：

```rust
fn len(s: &str) -> usize {
    s.len()
}
```

真实形式可以理解成：

```rust
fn len<'a>(s: &'a str) -> usize {
    s.len()
}
```

再看这个：

```rust
fn first(s: &str) -> &str {
    &s[0..1]
}
```

可以省略，因为只有一个引用参数。

但这个不能省略：

```rust
fn choose(x: &str, y: &str) -> &str {
    x
}
```

虽然人能看出来返回的是 `x`，但省略规则不会根据函数体推断。它只看函数签名。

所以要写：

```rust
fn choose<'a>(x: &'a str, _y: &str) -> &'a str {
    x
}
```

# 结构体中的引用

如果结构体保存引用，就必须写生命周期。

```rust
struct User<'a> {
    name: &'a str,
}

fn main() {
    let name = String::from("rekord");
    let user = User { name: &name };

    println!("{}", user.name);
}
```

`User<'a>` 表示：

> `User` 里面的 `name` 引用，不能比它引用的字符串活得更久。

如果没有生命周期标注，编译器不知道 `User` 中的引用和外部数据之间是什么关系。

注意：这并不是让 `name` 活得更久，而是约束 `User` 不能活得比 `name` 更久。

# `'static`

`'static` 表示引用可以存活整个程序运行期间。

最常见的是字符串字面量：

```rust
let s: &'static str = "hello";
```

字符串字面量直接存储在程序二进制中，所以它可以拥有 `'static` 生命周期。

但不要把 `'static` 理解成“永远不释放”。更准确地说：

> 这个引用在整个程序运行期间都是有效的。

如果一个值本来就不是全局长期存在，加 `'static` 也不会让它变成全局长期存在。

# 生命周期和所有权、借用的关系

三者关系可以这样理解：

- 所有权：值归谁负责。
- 借用：不拿走所有权，临时使用值。
- 生命周期：保证借用期间，被借用的值仍然有效。

生命周期不是独立存在的。

它是借用系统用来防止悬空引用的规则。

# 总结

生命周期可以先记住这几句话：

- 生命周期解决引用是否有效的问题。
- 引用不能比它指向的值活得更久。
- 生命周期标注不会延长任何值的生命。
- 生命周期标注只是描述引用之间的关系。
- 多数情况下编译器能推导，不需要手写。
- 多个引用参数并返回引用时，常常需要手写。
- 结构体保存引用时，必须标注生命周期。
- `'static` 表示引用在整个程序运行期间有效。

所有权回答“值归谁”。

借用回答“不拿走所有权能不能用”。

生命周期回答“引用能活多久”。

这三篇合在一起，基本就是 Rust 内存安全模型的基础。
